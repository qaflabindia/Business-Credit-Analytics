# Secure Design Document — Credit Analytics (O&G Business AR)

**Document type:** Security and Data Governance
**Status:** Active
**Version:** 2.0
**Owner:** Risk Analytics / Model Governance
**Covers:** Data classification, permitted uses, access controls, model governance, compliance

---

## 1. Executive summary

This document defines the security posture and governance requirements for the Credit Analytics synthetic O&G business AR dataset.

The core principle is: **synthetic does not mean unclassified and unrestricted**.

Although the dataset contains no real personal data, no real company data, and no real financial transactions, it is calibrated to resemble a real energy company's commercial credit portfolio in structure, scale, and behaviour. It therefore carries data governance obligations.

---

## 2. What this dataset is and is not

### 2.1 What it is

- A **fully synthetic** dataset generated by calibrated statistical models
- Designed for **internal model development, testing, analytics prototyping, and analyst training**
- Every number is generated from mathematical equations with known seeds
- No real person, company, transaction, or credit event is represented

### 2.2 What it is not

- Not a sample of, or derived from, any real credit portfolio or customer data
- Not a regulatory submission or regulatory-quality dataset
- Not a record of real financial transactions or real receivables
- Not approved for use in any real credit decision, limit-setting, or pricing process

## 3. Data classification

| Artefact | Recommended classification | Rationale |
|---|---|---|
| `data_out/*.parquet` | **Internal / Restricted** | Mimics real portfolio structure; must not be confused with real data |
| `credit_analytics/*.py` | **Internal** | Contains calibrated model parameters |
| `TECHNICAL_SPEC.md` | **Internal** | Documents AR ageing model equations |
| `DATA_DICTIONARY.md` | **Internal** | Could inform reverse-engineering of model structure |
| `README.md`, `GETTING_STARTED.md` | **Internal** | General purpose |
| ML models trained on this data | **Internal / Restricted** | Inherits classification of training data |

Do not classify this data as Public or Open.

## 4. Permitted and prohibited uses

### 4.1 Permitted uses

| Use | Notes |
|---|---|
| Prototyping AR monitoring dashboards | Primary intended use |
| Building and testing credit-hold trigger models | Primary intended use |
| Credit limit optimisation model development | Primary intended use |
| Stress testing AR exposure and bad-debt projections | Primary intended use |
| Training commercial credit analysts on realistic data | Recommended use |
| Unit testing and regression testing of credit analytics pipelines | Permitted |
| Academic research with appropriate disclosure that data is synthetic | Permitted with attribution |

### 4.2 Prohibited uses

| Use | Why prohibited |
|---|---|
| Real credit decisions about real customers | Not real data; no empirical validity |
| Setting real credit limits for real counterparties | Not real data |
| Regulatory capital calculation or reporting | Not a regulatory-quality dataset |
| Publication claiming this represents real market data | Misrepresentation |
| Training models to be deployed in production without re-validation on real data | Synthetic-to-real domain gap |
| Any use that violates applicable data protection or financial regulation | General prohibition |

## 5. Access controls

### 5.1 Recommended access boundaries

| Role | Access level | Justification |
|---|---|---|
| Credit analytics team | Read/write `data_out/` | Primary users |
| Model development team | Read/write `data_out/`, read source code | Need full access |
| Model validation team | Read `data_out/`, read documentation | Independent review |
| IT / infrastructure | No access to `data_out/` | No business need |
| External parties | No access without explicit approval | Third-party risk |

### 5.2 Labelling requirements

All exported files, presentations, and reports derived from this dataset must be clearly labelled:

> **"Synthetic data — not real customer or financial data. For internal model development and testing only."**

## 6. Model governance

### 6.1 Model inventory

This generator constitutes a **data generation model**. It must be entered into the institution's model inventory with:
- Model ID and owner
- Date of first use and last review
- Use case scope
- Known limitations (see Section 7)
- Link to this document and `TECHNICAL_SPEC.md`

### 6.2 Model review triggers

Review the generator and its outputs when any of the following occur:
- The institution's real AR portfolio composition changes materially
- New customer types are onboarded that are not represented in the 8 modelled types
- The macro calibration anchors (baseline Brent, GDP, BBB spreads) become significantly stale
- A validation check begins failing persistently
- The generator is used for a materially different use case than originally documented

### 6.3 Validation

Every generated dataset undergoes 8 automated validation checks (see `validation.py`). The full validation output must be reviewed by the user before using the data for any analytical purpose. Datasets with failed checks should not be used without investigation.

## 7. Known limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Synthetic-to-real gap | Models trained on synthetic data may not generalise directly to real customer behaviour | Always re-validate on real data before production deployment |
| Aggregate AR (not invoice-level) | Cannot support invoice-matching or dispute analytics | Use real invoice data for those use cases |
| Static customer ratings | Ratings do not migrate over time in the current implementation | Acceptable for most AR analytics; note limitation for rating-transition studies |
| Fixed customer population | New customers entering and exiting the portfolio are not modelled | Add scenario-specific customer churn if needed |
| Calibrated to Shell-like scale | May not represent smaller or more regional O&G companies accurately | Recalibrate parameters for different portfolio compositions |
| No individual invoice-level detail | Bad-debt provisioning at invoice level is not possible | Acceptable for portfolio-level stress testing |

---

## 8. Incident response

If the synthetic dataset is accidentally shared with an external party or mistakenly labelled as real data:

1. Immediately notify the data governance / model risk team
2. Retract the shared material and confirm deletion
3. Assess whether the recipient may have drawn incorrect conclusions about real credit exposures
4. Document the incident in the model risk log

Although no real data is involved, misleading an external party about the nature of the data (synthetic vs real) may have legal or regulatory implications and should be treated as a data governance incident.

---

## 9. Retention and disposal

| Artefact | Retention period | Disposal method |
|---|---|---|
| Generated `data_out/*.parquet` | Duration of the project or model development cycle | Secure deletion |
| Source code (`credit_analytics/`) | As long as the generator is in the model inventory | Archive with model inventory record |
| Documentation | As long as the generator is in use | Archive with model inventory record |

Generated parquet files should not be retained indefinitely. When a model development project is complete, the data should be deleted unless retained for reproducibility audit purposes.
