# Secure Design Document — Credit Analytics

**Document type:** Security and Data Governance
**Status:** Active
**Version:** 1.0
**Owner:** Risk Analytics / Model Governance
**Covers:** Data classification, permitted uses, access controls, model governance, compliance, incident response

---

## 1. Executive summary

This document defines the security posture and governance requirements for the Credit Analytics synthetic dataset. Because the dataset is specifically designed to resemble a real financial institution's credit portfolio in structure and calibration, it carries data governance obligations even though it contains no real personal data, no real company data, and no real financial data.

The core principle is: **synthetic does not mean unclassified and unrestricted**.

---

## 2. What this dataset is and is not

### 2.1 What it is

- A **fully synthetic** dataset generated by calibrated statistical models
- Calibrated to publicly available disclosures from Shell plc and the broader oil and gas credit market
- Designed for **internal model development, testing, validation, and training only**
- Every number is generated from mathematical equations with known seeds
- No real person, company, transaction, or credit event is represented

### 2.2 What it is not

- Not a sample of, or derived from, any real credit portfolio
- Not a regulatory submission or regulatory-quality dataset
- Not a record of real financial transactions
- Not a product of any proprietary Shell system or data feed
- Not approved for use in any real credit decision, pricing, or limit-setting process

---

## 3. Data classification

The synthetic dataset and its documentation should be classified according to the institution's data classification framework. The recommended classification is:

| Artefact | Recommended classification | Rationale |
|---|---|---|
| `data_out/*.parquet` | **Internal / Restricted** | Mimics real portfolio structure; must not be confused with real data |
| `credit_analytics/*.py` (source code) | **Internal** | Contains model parameters calibrated to public Shell disclosures |
| `TECHNICAL_SPEC.md` (equations) | **Internal** | Documents PD/LGD model implementation |
| `DATA_DICTIONARY.md` | **Internal** | Could inform reverse-engineering of model structure |
| `README.md`, `GETTING_STARTED.md` | **Internal** | General purpose; does not contain sensitive parameters |
| Trained ML models built on this data | **Internal / Restricted** | Inherits classification of the training data |

**Do not classify this data as Public or Open.** While the underlying calibration sources (Shell public disclosures, EIA, IMF) are public, the assembled synthetic dataset mimics proprietary portfolio structure and should be treated as internal.

## 4. Permitted and prohibited uses

### 4.1 Permitted uses

| Use | Notes |
|---|---|
| PD / LGD / ECL model development | Primary intended use |
| Credit scoring model prototyping | Primary intended use |
| IFRS 9 staging model development | Primary intended use |
| Stress testing framework development | Primary intended use |
| Credit surveillance dashboard prototyping | Primary intended use |
| Training credit analysts on realistic data | Primary intended use |
| Model validation benchmark testing | Primary intended use |
| Academic research (with appropriate disclosure) | Permitted provided the synthetic nature is disclosed |
| Teaching credit analytics methodology | Permitted provided students are informed of synthetic nature |

### 4.2 Prohibited uses

| Use | Reason |
|---|---|
| Making real credit decisions about any company | This data does not represent any real borrower |
| Filing regulatory returns or prudential submissions | This data does not meet regulatory data quality standards |
| Using borrower IDs or characteristics to infer real companies | No real company data is embedded — any similarity is coincidental |
| Publishing results that imply these are real defaults or recoveries | Would be misleading |
| Training AI/ML models deployed in production credit decisions without additional validation on real data | Synthetic data gap — real data validation is required before production deployment |
| Selling or licensing this dataset to third parties | Without explicit authorisation from the institution |
| Using this data in any legal proceeding as evidence of market practice | This is a calibrated synthetic, not an empirical market dataset |

## 5. Access controls

### 5.1 Minimum access requirements

| Role | Access level |
|---|---|
| Credit model developers | Read/write to data_out/, read-only to source code |
| Credit risk analysts | Read-only to data_out/ |
| Model validators | Read/write to data_out/, read-only to source code |
| External researchers (approved) | Read-only to data_out/ only — no source code access |
| Vendors / third parties | No access without explicit data-sharing agreement and DPA |
| External auditors | Read-only to data_out/ and documentation only |

### 5.2 Storage recommendations

- Store `data_out/` in an access-controlled internal file system or object store
- Do not store in public cloud buckets without encryption and IAM controls
- Do not email parquet files externally — use secure file transfer protocols
- If storing in a shared data platform, apply column-level or table-level access controls to prevent uncontrolled self-service access

### 5.3 Labelling requirement

Every parquet file and every notebook or dashboard that uses this data must include a visible label:

```
SYNTHETIC DATA — FOR INTERNAL DEVELOPMENT AND TESTING ONLY
This dataset contains no real company, transaction, or credit event data.
It must not be used for real credit decisions, regulatory submissions, or external publication.
```

## 6. Reproducibility as a security control

The deterministic seed architecture serves a dual purpose: it enables reproducibility for model development, and it also provides an audit trail:

- Any output can be regenerated exactly from the master seed and the source code version
- If the dataset is modified after generation (e.g. data manipulation), the modification can be detected by regenerating from seed and comparing
- The master seed (`MASTER_SEED = 20260605`) and all child seeds are documented in `config.py`

**Recommendation:** Record the exact git commit hash or source code version alongside any model trained on this data, to enable exact regeneration of the training set.

## 7. Model governance requirements

### 7.1 Model inventory

Any model trained on this synthetic dataset must be registered in the institution's model inventory with:

- Model name and version
- Training data: this synthetic dataset (version, scale, scenarios used)
- Intended use case (PD, LGD, scoring, etc.)
- Real-data validation status (required before production deployment)
- Model owner and validator

### 7.2 Real-data validation requirement

**A model trained entirely on synthetic data must not be deployed in a production credit process without validation on real observed data.**

The synthetic dataset can be used to:
- Develop the model architecture
- Test the model pipeline end-to-end
- Validate model behaviour qualitatively
- Benchmark against known ground truth

It cannot substitute for:
- Long-run default rate calibration (requires real historical default data)
- Regulatory IRB model validation (requires real obligor data)
- IFRS 9 ECL model sign-off (requires real loan book data)
- Backtesting against real outcomes

### 7.3 Documentation requirements

Any model trained on this data must document in its model specification:

1. That training data is synthetic
2. The specific version and seed used for the synthetic dataset
3. The results of qualitative and quantitative validation against real data
4. Known differences between synthetic data characteristics and real portfolio characteristics

### 7.4 Effective challenge

Model validators reviewing models trained on this data should test:
- Whether the model's PD rank ordering is preserved when applied to real borrowers
- Whether the model's LGD estimates are within regulatory guidance bounds
- Whether the stress scenario sensitivities are plausible compared to real historical stress periods
- Whether the model has been biased by synthetic data artefacts (e.g. overfitting to the synthetic segment structure)

## 8. Data quality and integrity controls

### 8.1 Automated validation (built-in)

Every generation run executes 11 correctness checks. A generation run should not be used if any check fails:

| Check | What a failure means |
|---|---|
| `accounting_identities.net_debt` | Balance sheet is inconsistent — do not use for financial ratio models |
| `rating_monotonicity` | PD model is not correctly calibrated |
| `pd_lgd_dependence` | Stress sensitivity is absent — stress testing will be misleading |
| `seniority_ordering` | Recovery model is incorrectly calibrated |
| `no_future_leakage` | Time series contains look-ahead bias — all time-series models will be invalid |

### 8.2 Version control

- The source code generating the dataset should be under version control (git)
- Dataset generation runs should be logged with: date, git commit, scale, seed, scenarios, validation pass/fail
- If the source code changes and the dataset is regenerated, the new dataset should be treated as a new version

### 8.3 Data lineage

The generation pipeline is fully traceable:
- Every column's value can be traced to a specific equation in `TECHNICAL_SPEC.md`
- Every random draw can be traced to a specific seed in `config.py`
- The validation suite provides automated evidence of data integrity

## 9. AI and large-language model use restrictions

### 9.1 Using this data to train LLMs or foundation models

Training a general-purpose LLM or foundation model on this synthetic credit dataset is **not recommended** without the following controls:

- The model must not be deployed in a context where users might confuse synthetic credit outcomes with real market data
- Training data provenance must be disclosed in the model card
- The model must not be used to generate synthetic credit data in a way that amplifies the original calibration errors

### 9.2 Using LLMs to analyse this data

LLMs used to query or analyse this data in an analyst-assist context should be instructed:

```
This data is synthetic credit data for training and testing purposes.
It does not represent real companies, real loans, or real defaults.
Do not treat any output from this data as evidence about real credit markets.
```

### 9.3 Agentic use

If this dataset is used in an agentic credit analytics pipeline (automated model training, automated report generation, automated credit recommendations), the following controls apply:

- The agent must not be given authority to make real credit decisions
- Any output labelled as a credit recommendation must include a synthetic-data disclaimer
- Human review is required before any output is presented to a real credit officer as actionable

## 10. Regulatory and compliance considerations

### 10.1 Prudential regulations (Basel IV, CRR3)

This synthetic dataset does not satisfy the data quality requirements for Internal Ratings Based (IRB) approach model validation under Basel IV / CRR3:
- IRB models require real historical default data with a minimum 5-year observation period
- LGD models require real workout data with economic downturn calibration
- This dataset can be used for model development and preliminary validation only

### 10.2 IFRS 9

Models trained on this dataset cannot be used to generate auditable IFRS 9 ECL estimates without:
- Validation on real loan book data
- Sign-off from the institution's independent model validation team
- Documentation of the bridge from synthetic training to real application

### 10.3 GDPR and data privacy

This dataset contains no personal data. No individual persons are represented. GDPR obligations do not apply to the dataset itself.

However, if this dataset is used in a pipeline that also processes real personal or financial data, standard GDPR controls apply to that real-data portion.

### 10.4 SR 11-7 / Model Risk Management (US banks)

For US institutions subject to SR 11-7 model risk guidance:
- This dataset is a **model input** and must be documented in the model inventory
- The generation methodology must be documented (this technical specification satisfies that requirement)
- Effective challenge of the synthetic generation assumptions is required before production use

## 11. Incident response

If the synthetic dataset is:

**Accidentally published externally or shared with an unauthorised party:**
1. Notify the data owner and information security immediately
2. Assess whether any real data was inadvertently included (it should not be, but confirm by regenerating from seed and comparing)
3. Issue a retraction or clarification notice if the data was published in a way that could be confused with real portfolio data
4. Review access controls and storage configuration

**Discovered to contain an error in the generation model:**
1. Document the error and its impact on derived analyses
2. Regenerate the dataset using the corrected code with a new version number
3. Notify all model developers who used the erroneous version
4. Re-validate any models trained on the erroneous version

**Used to make a real credit decision (prohibited use):**
1. Immediately reverse or halt the decision pending review
2. Re-evaluate using real data
3. Conduct a root-cause analysis of how the prohibited use occurred
4. Implement additional access controls or labelling to prevent recurrence

---

## 12. Responsible disclosure

If a researcher or analyst identifies a material error in the calibration or a case where the synthetic data could be mistaken for real market data, they should report it to the model owner. This dataset is a tool for internal analytics; its trustworthiness depends on maintaining accurate documentation of its limitations.
